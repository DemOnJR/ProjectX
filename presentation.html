First-Time GCP Setup

Complete these steps once when setting up a brand-new Google Cloud project. Everything is in order — run them top to bottom.

Step 0 — Install Required Tools

ToolInstallVersion
gcloudcloud.google.com/sdklatest
terraformdeveloper.hashicorp.com/terraform≥ 1.5.0
dockerdocs.docker.comany recent
kubectlkubernetes.ioany recent

Step 1 — Log in to gcloud

# Authenticate your Google account with gcloud
gcloud auth login

# Verify the correct account is active
gcloud auth list

Step 2 — Create the GCP Project

export PROJECT_ID="my-projectx-123456"   # choose a unique project ID

gcloud projects create $PROJECT_ID --name="ProjectX"

# Confirm it was created
gcloud projects describe $PROJECT_ID

Project IDs must be globally unique across all of Google Cloud, 6–30 characters, lowercase letters/numbers/hyphens only.

Step 3 — Link a Billing Account

# List your billing accounts to find the ID
gcloud billing accounts list

# Link the billing account to the project
gcloud billing projects link $PROJECT_ID \
  --billing-account=XXXXXX-XXXXXX-XXXXXX

GKE cannot be created without an active billing account. You will get a FAILED_PRECONDITION error if billing is not linked.

Step 4 — Set the Active Project

gcloud config set project $PROJECT_ID

Step 5 — Set Application Default Credentials (for Terraform)

gcloud auth application-default login

Terraform uses Application Default Credentials (ADC) — not your gcloud auth login session. This separate command is required. Do not set GOOGLE_APPLICATION_CREDENTIALS unless you intentionally use a service account key file.

Step 6 — Grant Your Account the Required IAM Roles

export ACCOUNT="your-email@gmail.com"

gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="user:$ACCOUNT" --role="roles/owner"

Minimal roles if you cannot use roles/owner: roles/container.admin, roles/compute.networkAdmin, roles/artifactregistry.admin, roles/iam.serviceAccountAdmin, roles/resourcemanager.projectIamAdmin, roles/serviceusage.serviceUsageAdmin.

Step 7 — Enable Service Usage API (if not already on)

This API is required for Terraform to enable other APIs. It is on by default for new projects but worth confirming:

gcloud services enable serviceusage.googleapis.com --project=$PROJECT_ID

Step 8 — (Optional) Pre-enable GCP APIs

Terraform does this automatically when manage_project_services = true. Run manually only if your account lacks Service Usage permissions:

gcloud services enable \
  artifactregistry.googleapis.com \
  compute.googleapis.com \
  container.googleapis.com \
  iam.googleapis.com \
  --project=$PROJECT_ID

All done. You can now continue to the Deploy section below.

Deploy with Terraform

Fork or clone both repos so you have your own copies to push changes to.

git clone https://github.com/YOUR_USERNAME/ProjectX.git
git clone https://github.com/YOUR_USERNAME/ProjectX-ArgoCD.git

Create your variables file from the example template.

cd ProjectX/infra/terraform
cp terraform.tfvars.example terraform.tfvars

Edit terraform.tfvars and set at minimum:

gcp_project_id = "YOUR_PROJECT_ID"
gitops_repo_url = "https://github.com/YOUR_USERNAME/ProjectX-ArgoCD.git"

terraform.tfvars is git-ignored so your credentials stay local.

Initialise Terraform — downloads providers and modules.

terraform init

Review the plan — verify what will be created before spending money.

terraform plan

Apply — this provisions the full stack (~15–20 min first time).

terraform apply

After apply, Terraform prints useful outputs: cluster credentials command, Artifact Registry URL, Argo CD access command, and initial password command.

Configure kubectl — run the command printed by Terraform output get_credentials_command:

gcloud container clusters get-credentials projectx-gke \
  --region europe-west3 --project YOUR_PROJECT_ID

What to do next (in order):
1. Build & push the Docker image (next section)
2. Set up Cloudflare Tunnels if you want public access
3. Set up Vault integration if you want centralized secret management
4. Access Argo CD to verify the app is running

Build & Push the Docker Image

The GKE cluster pulls images from Artifact Registry. Build and push the image after terraform apply completes (or whenever you update the app).

export PROJECT_ID="YOUR_PROJECT_ID"
export REGION="europe-west3"
export IMAGE="$REGION-docker.pkg.dev/$PROJECT_ID/projectx/projectx-api:0.1.0"

# Authorise Docker to push to Artifact Registry
gcloud auth configure-docker "$REGION-docker.pkg.dev"

# Build from the app directory
docker build -t "$IMAGE" ./app

# Push
docker push "$IMAGE"

After pushing a new image tag, update image.tag in ProjectX-ArgoCD/apps/projectx-api/values.yaml and commit/push. Argo CD will roll out the new version automatically within ~3 minutes.

Run Locally (no Docker)

cd ProjectX/app
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8000
# Open http://localhost:8000

Accessing Argo CD

Get the initial admin password:

kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d && echo

Forward the Argo CD port:

kubectl -n argocd port-forward svc/argocd-server 8080:80

Open http://localhost:8080 and log in with username admin.

Verify the Application

# Check Argo CD application status
kubectl get application -n argocd

# Check pods in the projectx namespace
kubectl get pods -n projectx

# Check all resources deployed by Argo CD
kubectl get all -n projectx

How Argo CD Syncs

Argo CD is configured with automated sync (prune: true, selfHeal: true):

  • prune — deletes K8s resources that are removed from the Helm chart.
  • selfHeal — reverts manual changes to the cluster to match the GitOps repo.
  • CreateNamespace — creates the projectx namespace automatically on first sync.

Force a Manual Sync

argocd app sync projectx-api
# or via kubectl
kubectl patch application projectx-api -n argocd \
  --type merge -p '{"operation":{"sync":{}}}'

Optional Features

Cloudflare Tunnels — Public Access Without a Load Balancer

Cloudflare tunnels let you expose both Argo CD and the ProjectX API on public hostnames without opening firewall ports or paying for GCP Load Balancers. You create two tunnels in the Cloudflare dashboard and paste the tokens into terraform.tfvars — Terraform handles the rest.

Argo CD

argocd.pbcv.dev

Managed by Terraform directly — a cloudflared Deployment is created in the argocd namespace pointing at argocd-server:80.

ProjectX API

projectx.pbcv.dev

Managed by Argo CD via the Helm chart. Terraform injects the secret; the chart deploys the tunnel pod when cloudflareTunnel.enabled: true.

Step 1 — Create Two Tunnels in Cloudflare

Go to one.dash.cloudflare.com → Networks → Tunnels → Create a tunnel. Choose Cloudflared as the connector type.

Create the first tunnel (for Argo CD):

  • Name: argocd-projectx
  • Public hostname: argocd.pbcv.dev → Service: http://argocd-server.argocd.svc.cluster.local:80

Copy the tunnel token — you'll use it as argocd_tunnel_token.

Create the second tunnel (for the API):

  • Name: projectx-api
  • Public hostname: projectx.pbcv.dev → Service: http://projectx-api.projectx.svc.cluster.local:80

Copy the tunnel token — you'll use it as projectx_tunnel_token.

You said you'll add the DNS records yourself (argocd.pbcv.dev and projectx.pbcv.dev) in the Cloudflare dashboard. The tunnel connector will also create them automatically when you configure the public hostname above.

Step 2 — Add Tokens to terraform.tfvars

argocd_tunnel_token   = "eyJhIjoiNmZmM2Zm..."
projectx_tunnel_token = "eyJhIjoiOTk5OTk5..."

terraform.tfvars is git-ignored. Your tokens will never be committed to source control.

Step 3 — Enable the Tunnel in values.yaml (ProjectX API only)

Edit ProjectX-ArgoCD/apps/projectx-api/values.yaml and set:

cloudflareTunnel:
  enabled: true
  hostname: projectx.pbcv.dev

Commit and push — Argo CD will roll out the cloudflared pod automatically. The Argo CD tunnel is fully managed by Terraform and needs no changes in values.yaml.

Step 4 — Apply

cd ProjectX/infra/terraform
terraform apply

Terraform creates:

  • A cloudflared Deployment (2 replicas) in the argocd namespace pointing at argocd-server:80
  • A cloudflared-token Secret in the projectx namespace (used by the Helm chart)

After apply, https://argocd.pbcv.dev and https://projectx.pbcv.dev will be live within ~1 minute (time for cloudflared to register the tunnel).

Vault Secrets Operator (Advanced — Optional)

Automatically syncs secrets from HashiCorp Vault into Kubernetes secrets. Useful if you want to rotate the Cloudflare tunnel token centrally without touching the cluster.

Requires a running Vault instance with the Kubernetes auth method configured for the projectx-api role.

Enable in terraform.tfvars:

install_vault_secrets_operator = true

Enable in values.yaml:

vault:
  enabled: true
  address: "https://vault.pbcv.dev"
  kubernetesAuthRole: projectx-api

GCP Load Balancer Ingress (Alternative)

If you prefer a GCP HTTPS Load Balancer instead of Cloudflare Tunnel for the API:

ingress:
  enabled: true
  className: gce
  host: projectx.pbcv.dev

GCP Load Balancers cost ~$18/month. Disable the Ingress and push the change so Argo CD removes the GCP LB before running terraform destroy, otherwise the LB remains as an orphaned resource.

CI — Automated Build & Deploy

When you push code to main, GitHub Actions automatically builds the Docker image, pushes it to Artifact Registry, and updates the image tag in the GitOps repo. Argo CD detects the commit and rolls out the new version — no manual steps.

Full automated flow

git push to
ProjectX/main → GitHub Actions
triggers → docker build
+ push to AR → updates image.tag
in ProjectX-ArgoCD → Argo CD detects
commit → rolls out new
image to GKE ✓

The workflow only triggers when files inside app/ change — infra or chart changes do not trigger a build.

Authentication — No Keys Required

GitHub Actions authenticates to GCP using Workload Identity Federation (WIF). GitHub's OIDC token is exchanged for a short-lived GCP token at runtime. No service account JSON keys are ever created or stored.

Terraform creates the WIF pool, provider, CI service account, and IAM bindings automatically when you set github_repo in terraform.tfvars.

Step 1 — Set github_repo in terraform.tfvars

In infra/terraform/terraform.tfvars:

github_repo = "DemOnJR/ProjectX"

Then apply:

cd ProjectX/infra/terraform
terraform apply

After apply, note the two new outputs — you'll need them in the next steps:

terraform output ci_wif_provider
terraform output ci_service_account

Step 2 — Create a GitOps PAT on GitHub

The CI workflow needs to push commits to ProjectX-ArgoCD. Create a Personal Access Token:

Go to github.com/settings/tokens/new (Classic token).

Give it a name like projectx-gitops-ci, set expiration, and tick the repo scope (full repository access).

Copy the token — you will only see it once.

Step 3 — Add GitHub Actions Variables & Secret

Go to your ProjectX repo → Settings → Secrets and variables → Actions.

Add these as Variables (not secrets — they are not sensitive):

Variable nameValue (from Terraform outputs)
WIF_PROVIDERvalue of terraform output ci_wif_provider
GCP_SERVICE_ACCOUNTvalue of terraform output ci_service_account
GCP_PROJECT_IDyour GCP project ID (e.g. gcp-projectx-496112)

Add this as a Secret:

Secret nameValue
GITOPS_PATthe GitHub PAT you created in Step 2

Step 4 — Push and Watch It Work

Edit anything inside app/ (e.g. the hello message in main.py) and push to main:

git add app/main.py
git commit -m "feat: update hello message"
git push origin main

Then watch:

# GitHub Actions tab in your repo — watch the workflow run
# After it finishes, check the new commit in ProjectX-ArgoCD:
#   https://github.com/DemOnJR/ProjectX-ArgoCD/commits/main

# Argo CD will sync within ~3 minutes, or force it:
kubectl patch application projectx-api -n argocd \
  --type merge -p '{"operation":{"sync":{}}}'

# Watch the rollout
kubectl rollout status deployment/projectx-api -n projectx

What the Workflow Does (ci.yml)

on:
  push:
    branches: [main]
    paths: ["app/**"]        # only triggers when app code changes

steps:
  1. Checkout code
  2. Authenticate to GCP via WIF (no keys!)
  3. docker build + push  → AR tag = first 8 chars of git SHA
  4. Clone ProjectX-ArgoCD
  5. yq: update .image.tag in values.yaml
  6. git commit + push     → ArgoCD detects and deploys

HashiCorp Vault Integration

Run this after terraform apply in infra/terraform/ completes. The vault Terraform reads the live GKE cluster to configure Kubernetes auth.

All your secrets live in one gitignored file — infra/terraform-vault/terraform.tfvars. Run terraform apply there and Terraform writes everything to Vault automatically. The cluster then reads secrets from Vault via the Vault Secrets Operator (VSO) — no tokens ever touch Git.

How It Works End-to-End

terraform.tfvars
(your secrets file) → terraform apply
(terraform-vault/) → Vault
kv/projectx, kv/argocd → VSO reads Vault
(Kubernetes auth) → K8s Secret created
cloudflared-token → cloudflared starts ✓

VSO also refreshes secrets automatically every hour. Rotate the token in Vault → pods get the new value without any restart.

What Terraform Creates in Vault

Vault side
  • Kubernetes auth backend at kubernetes/
  • Configured with GKE API endpoint + CA cert
  • Policy projectx-api → read kv/data/projectx
  • Role projectx-api → SA projectx-vault-auth in namespace projectx
  • Secret kv/projectx → { token: "..." }
  • Secret kv/argocd → { token: "..." }
Cluster side (ArgoCD)
  • ServiceAccount projectx-vault-auth
  • VaultConnection → vault.pbcv.dev
  • VaultAuth → Kubernetes auth, role projectx-api
  • VaultStaticSecret → kv/projectx.token → K8s Secret cloudflared-token

Step 1 — Prerequisites: KV v2 Engine

Your Vault must have the KV v2 secrets engine enabled at kv/. Check and enable if needed:

vault secrets list

# If kv/ is not listed:
vault secrets enable -path=kv kv-v2

Step 2 — Enable VSO in Main Terraform

Make sure VSO is installed by the main Terraform. In infra/terraform/terraform.tfvars:

install_vault_secrets_operator = true

If you already ran terraform apply without it, just run it again — it only adds the VSO Helm release.

Step 3 — Get Your Vault Token

# Login with username/password (or whichever auth method you use)
vault login -method=userpass username=YOUR_USERNAME

# Verify
vault token lookup

Step 4 — Fill in the Secrets File

This is the only file you need to edit. It is gitignored — your tokens never go to Git.

cd ProjectX/infra/terraform-vault
cp terraform.tfvars.example terraform.tfvars

Open terraform.tfvars and fill it in:

gcp_project_id = "YOUR_PROJECT_ID"

vault_address = "https://vault.pbcv.dev"

# Cloudflare tunnel tokens — paste both here.
# Terraform writes them to Vault. Leave empty to skip (if already in Vault).
cloudflare_projectx_token = "eyJ..."   # → kv/projectx  (used by VSO for the API)
cloudflare_argocd_token   = "eyJ..."   # → kv/argocd    (stored for reference)

You do not need to touch infra/terraform/terraform.tfvars for secrets anymore. Remove projectx_tunnel_token and argocd_tunnel_token from it if you set them before — VSO now handles secret injection.

Step 5 — Apply the Vault Terraform

cd ProjectX/infra/terraform-vault
terraform init
terraform apply

Terraform will:

  • Enable the Kubernetes auth backend in Vault and point it at your GKE cluster
  • Create policy projectx-api and role projectx-api
  • Write kv/projectx { token: "..." } and kv/argocd { token: "..." } to Vault

Want to add more secrets later? Add variables to terraform.tfvars and new vault_kv_secret_v2 resources to main.tf. Rerun terraform apply — Vault is updated automatically.

Step 6 — Enable Vault + Cloudflare Tunnel in values.yaml

Edit ProjectX-ArgoCD/apps/projectx-api/values.yaml and set:

cloudflareTunnel:
  enabled: true
  hostname: projectx.pbcv.dev

vault:
  enabled: true
  address: "https://vault.pbcv.dev"
  kubernetesAuthMount: kubernetes
  kubernetesAuthRole: projectx-api

Commit and push. Argo CD syncs automatically — within ~2 minutes VSO creates the cloudflared-token K8s secret and the cloudflared pods start.

Step 7 — Verify

# Check VSO synced the secret
kubectl get secret cloudflared-token -n projectx

# Check cloudflared pods are running
kubectl get pods -n projectx

# Check VSO logs if something is wrong
kubectl logs -n vault-secrets-operator-system \
  -l app.kubernetes.io/name=vault-secrets-operator | tail -30

Rotating a Secret

Update the value in Vault — VSO picks it up automatically within refreshAfter (default 1 hour). To force immediate refresh:

# Update in Vault
vault kv put kv/projectx token="eyJ_NEW_TOKEN..."

# Force VSO to re-sync now
kubectl annotate vaultstaticsecret projectx-api-cloudflared-token \
  vault.hashicorp.com/requestReconcile="$(date)" -n projectx --overwrite

Cleanup

When you destroy the cluster, clean up Vault first:

# 1. Remove Vault configuration (auth backend, role, policy)
cd ProjectX/infra/terraform-vault
terraform destroy

# 2. Then destroy the cluster
cd ../terraform
terraform destroy

How Kubernetes Auth Works (Internals)

  1. VSO presents the projectx-vault-auth service account JWT to Vault at /v1/auth/kubernetes/login
  2. Vault verifies the JWT against the GKE API server using the CA cert configured by Terraform
  3. Vault checks that the SA name and namespace match the projectx-api role
  4. Vault issues a short-lived token scoped to the projectx-api policy
  5. VSO reads kv/data/projectx and creates/updates the cloudflared-token K8s secret
  6. Cloudflared pod reads the secret and connects to Cloudflare ✓

Full Cleanup — terraform destroy

Running terraform destroy removes everything Terraform created. Here is what happens step by step:

ResourceHow it's cleaned up
K8s resources (projectx namespace)Deleted when the GKE cluster is deleted
Argo CD Helm releaseTerraform destroys it before the cluster
Docker images in Artifact RegistryDestroy-time provisioner empties the repo first
Artifact Registry repositoryDestroyed after images are removed
GKE node pool & clusterDestroyed by the GKE Terraform module
GKE node service accountDestroyed by the GKE module
VPC subnets & networkDestroyed by the network module
GCP project APIsLeft enabled (disable_on_destroy = false)

Before destroying: if you enabled the GCP Load Balancer Ingress (ingress.enabled: true), disable it in values.yaml and push the change so Argo CD removes the GCP LB first. Otherwise the LB will remain in GCP and continue to incur costs.

cd ProjectX/infra/terraform
terraform destroy

If terraform destroy fails mid-way, you can rerun it safely — Terraform is idempotent for destroy operations. You can also delete the GCP project entirely from the console to guarantee zero remaining resources.

Delete the GCP Project (nuclear option)

gcloud projects delete YOUR_PROJECT_ID

This permanently deletes the project and all resources inside it. There is a 30-day grace period during which you can restore it.

Troubleshooting

Pods stuck in ErrImagePull / ImagePullBackOff

Cause: GKE nodes cannot pull from Artifact Registry.

Fix: Already handled — Terraform now grants roles/artifactregistry.reader to the GKE node service account automatically.

Verify:

kubectl describe pod -n projectx -l app.kubernetes.io/name=projectx-api | grep -A5 Events

The image URL must match the repository URL in values.yaml exactly.

Cloudflared pod stuck in CrashLoopBackOff

Cause: The cloudflared-token secret does not exist in the projectx namespace.

Fix: Create it manually:

kubectl create secret generic cloudflared-token \
  --from-literal=token=YOUR_TUNNEL_TOKEN -n projectx

Or set cloudflareTunnel.enabled: false in values.yaml if you don't need the tunnel.

Terraform apply fails with "master_authorized_networks"

Cause: Your current IP changed since the last apply (e.g. new network connection).

Fix: Rerun terraform apply — it detects your current IP via checkip.amazonaws.com and updates the allowlist.

Terraform destroy fails on Artifact Registry

Cause: Images existed in the repository before the destroy-time cleanup was in place.

Fix: Delete images manually then rerun destroy:

REPO="europe-west3-docker.pkg.dev/YOUR_PROJECT_ID/projectx"
gcloud artifacts docker images list "$REPO" \
  --format="value(DIGEST)" | sort -u | \
  while read digest; do
    gcloud artifacts docker images delete "$REPO@$digest" \
      --delete-tags --quiet
  done
terraform destroy

Argo CD Application stuck in OutOfSync

Check the Application events and logs:

kubectl describe application projectx-api -n argocd
kubectl logs -n argocd -l app.kubernetes.io/name=argocd-application-controller | tail -50

Common causes: GitOps repo URL typo, incorrect gitops_app_path, or VaultStaticSecret CRDs missing (enable VSO or set vault.enabled: false).

VSO pod running but cloudflared-token secret not created

Check the VaultStaticSecret status:

kubectl describe vaultstaticsecret -n projectx
kubectl logs -n vault-secrets-operator-system \
  -l app.kubernetes.io/name=vault-secrets-operator | tail -30

Common causes:

  • Auth backend not configured — run terraform apply in infra/terraform-vault/
  • Wrong role/SA name — verify vault_kubernetes_auth_backend_role.projectx_api matches values.yaml vault.serviceAccountName
  • Secret not in Vault yet — run vault kv put kv/projectx token="..."
  • KV v2 not enabled — run vault secrets enable -path=kv kv-v2

kubectl: "no cluster" or stale credentials

gcloud container clusters get-credentials projectx-gke \
  --region europe-west3 --project YOUR_PROJECT_ID
