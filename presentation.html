Technologies — What They Are & What They Do Here

A short explanation of each tool: why it was created, what problem it solves, and its specific role in this project.

🐳

Docker

Containerisation

Why it was invented: Before containers, deploying an app meant installing the right version of Python (or Java, Node…) on every server, dealing with conflicting dependencies, and facing the classic "it works on my machine" problem. Docker (2013) packages the app together with its entire runtime — OS libraries, language runtime, dependencies — into a single portable image. That image runs identically everywhere: a developer's laptop, a CI runner, a cloud VM, or a Kubernetes pod.

In this project: The FastAPI app lives in app/Dockerfile. It starts from python:3.12-slim, installs fastapi + uvicorn, and copies main.py. CI builds this image and pushes it to Artifact Registry on every code change. GKE then pulls and runs it.

☸️

Kubernetes (K8s)

Container Orchestration

Why it was invented: Running one container on one machine is easy. Running hundreds of containers across tens of machines — keeping them healthy, restarting crashed ones, distributing traffic, scaling up on load and down to save money — is not. Google ran everything in containers internally for years using a system called Borg. Kubernetes (2014) is the open-source version of those ideas. It is a cluster operating system: you declare what you want ("run 3 replicas of this container") and Kubernetes makes it happen and keeps it that way.

In this project: Kubernetes runs on GKE and hosts all workloads. It manages the projectx-api Deployment (keeps 2 pods running, health-checks them via /healthz), the HPA (scales from 2 to 10 replicas based on CPU), the ClusterIP Service (internal load balancing), and the cloudflared tunnel pods. It also runs ArgoCD and the Vault Secrets Operator.

☁️

GKE — Google Kubernetes Engine

Managed Kubernetes

Why it was invented: Running Kubernetes yourself means managing the control plane (the API server, scheduler, etcd database) — upgrades, HA, backups, certificates. That is a full-time job. Managed Kubernetes services (GKE, EKS, AKS) take over the control plane completely. You only manage the worker nodes (and in Autopilot mode, not even those). GKE (2014, Google's own product built on the same team that wrote Kubernetes) is considered the most mature of the managed offerings.

In this project: GKE hosts the cluster in europe-west3 across two zones. Terraform provisions it via the terraform-google-modules/kubernetes-engine module. It handles node upgrades, auto-repair, and auto-scaling transparently. The node pool uses e2-standard-2 machines and scales from 1 to 4 nodes per zone.

🏗️

Terraform

Infrastructure as Code

Why it was invented: Clicking through cloud consoles to create infrastructure is slow, error-prone, and impossible to reproduce. HashiCorp created Terraform (2014) so infrastructure can be described in code (.tf files), version-controlled in Git, reviewed in pull requests, and applied repeatedly to get the same result every time. It also tracks what it created in a state file, so it knows exactly what to change or delete.

In this project: infra/terraform/ provisions everything in GCP: the VPC, GKE cluster, Artifact Registry, IAM bindings, ArgoCD and VSO Helm releases, Cloudflare tunnel secrets, and the Workload Identity Federation pool for CI. infra/terraform-vault/ configures the Kubernetes auth backend in Vault after the cluster exists. One terraform apply creates the entire stack from nothing.

⎈

Helm

Kubernetes Package Manager

Why it was invented: A real Kubernetes app is not one YAML file — it is a Deployment, Service, ConfigMap, Ingress, HPA, ServiceAccount… all linked together. Copy-pasting and editing raw YAML for each environment (dev, staging, prod) is painful. Helm (2015) is the package manager for Kubernetes: it bundles all the YAML templates for an app into a chart, and you customise it by changing values in a single values.yaml file. It also tracks what was deployed and enables clean upgrades and rollbacks.

In this project: ProjectX-ArgoCD/apps/projectx-api/ is a Helm chart. Templates render the Deployment, Service, HPA, Ingress, cloudflared Deployment, and Vault CRDs — all driven by values.yaml. ArgoCD, VSO, and the argocd-apps application set are also installed as Helm releases managed by Terraform.

📦

Google Artifact Registry

Container Registry

Why it was invented: Docker images need to be stored somewhere accessible to the cluster — a registry. Docker Hub is public. Private registries let you control who can push or pull. Google Artifact Registry (2021, successor to Container Registry) is a fully managed private registry inside GCP, with tight IAM integration, vulnerability scanning, and zero egress cost to GKE in the same region.

In this project: Stores the projectx-api Docker images at europe-west3-docker.pkg.dev/gcp-projectx-496112/projectx/. CI pushes images tagged with the git SHA (e.g. :a1b2c3d4). GKE nodes pull from it using the node service account which Terraform grants roles/artifactregistry.reader. The repository is destroyed cleanly by a Terraform destroy-time provisioner that empties all images first.

⚙️

GitHub Actions

CI/CD

Why it was invented: Every software team needs Continuous Integration: automatically run tests, build artifacts, and deploy when code changes — no manual steps. GitHub Actions (2019) is GitHub's built-in CI/CD system. Instead of managing a separate Jenkins server or paying for a standalone CI SaaS, workflows live in .github/workflows/ right next to the code. Each workflow is triggered by Git events (push, PR, tag) and runs in disposable virtual machines.

In this project: .github/workflows/ci.yml triggers on every push to main that touches app/**. It authenticates to GCP via WIF (no stored keys), builds the Docker image, pushes it to Artifact Registry with a short-SHA tag, then clones ProjectX-ArgoCD and commits an updated image.tag. ArgoCD picks up that commit and deploys. Total time: ~3–5 minutes.

🔑

Workload Identity Federation (WIF)

Keyless Auth

Why it was invented: The classic way to let CI push to a cloud is to download a service account JSON key and store it as a CI secret. Keys are long-lived, can be leaked in logs, and are a pain to rotate. WIF (Google, 2021) lets an external identity — like a GitHub Actions OIDC token — be exchanged for a short-lived GCP token at job runtime. No key file is ever created. The token expires when the job ends.

In this project: Terraform creates a WIF pool (github-actions) with an attribute condition that only accepts tokens from DemOnJR/ProjectX. GitHub Actions exchanges its OIDC JWT for a GCP token that can impersonate the projectx-ci service account, which has roles/artifactregistry.writer. Nothing is stored as a secret — just three GitHub Actions variables that are not even sensitive.

🔄

Argo CD

GitOps Delivery

Why it was invented: Traditionally, CI pipelines do both building and deploying: kubectl apply is called at the end of the pipeline. This means the cluster state is only known at deploy time and there is no single source of truth for "what is actually running". GitOps (coined by Weaveworks, 2017) flips this: a Git repository is the desired state of the cluster, and a controller running inside Kubernetes continuously reconciles the cluster to match it. Argo CD (2018, CNCF graduated) is the most popular GitOps controller. It watches a repo, detects drift, and syncs — automatically or on demand.

In this project: ArgoCD runs in the argocd namespace and watches github.com/DemOnJR/ProjectX-ArgoCD. When CI commits a new image.tag, ArgoCD detects the change within ~3 minutes and performs a rolling update of the projectx-api Deployment. It also has prune: true (removes deleted resources) and selfHeal: true (reverts manual kubectl changes).

🌐

Cloudflare Tunnel

Secure Ingress

Why it was invented: Exposing a service to the internet traditionally requires a public IP, an open port, a load balancer, TLS certificates, and DDoS protection — all of which cost money and require ongoing management. Cloudflare Tunnel (formerly Argo Tunnel, 2018) works the other way: a small daemon (cloudflared) runs inside your network and makes an outbound connection to Cloudflare's edge. Cloudflare proxies inbound HTTPS traffic through that connection to your service. No public IP, no open ports, TLS managed by Cloudflare, DDoS protection included.

In this project: Two tunnels expose services publicly. The ArgoCD tunnel (argocd.pbcv.dev) is managed by Terraform as a Deployment in the argocd namespace, forwarding to argocd-server:80. The API tunnel (projectx.pbcv.dev) is managed by the Helm chart in the projectx namespace, forwarding to projectx-api:80. Both use tokens stored either directly in Terraform or via Vault + VSO.

🔐

HashiCorp Vault

Secrets Management

Why it was invented: Applications need secrets (passwords, API keys, tokens) to function. Storing them in environment variables, config files, or Git is insecure. HashiCorp Vault (2015) is a dedicated secret store: it encrypts secrets at rest, controls who can read what via policies, produces audit logs for every access, and can automatically rotate secrets. It also acts as a certificate authority and can generate short-lived database passwords on demand — so secrets are never long-lived.

In this project: Vault at vault.pbcv.dev stores the Cloudflare tunnel tokens (kv/projectx, kv/argocd). The Kubernetes auth backend (configured by infra/terraform-vault/) lets GKE pods prove their identity using their service account JWT. The Vault Secrets Operator reads the token from Vault and creates a cloudflared-token Kubernetes secret — refreshing it every hour without any human intervention or pod restart.

🤖

Vault Secrets Operator (VSO)

Secret Sync

Why it was invented: Vault is great, but apps running in Kubernetes still need secrets as Kubernetes Secrets or environment variables — they don't speak the Vault API natively. You could write a sidecar or an init container to fetch secrets on startup, but those are fragile and don't handle rotation. VSO (HashiCorp, 2023) is a Kubernetes operator: it watches VaultStaticSecret custom resources, authenticates with Vault on behalf of the workload, and keeps the corresponding Kubernetes Secret in sync automatically.

In this project: VSO is installed as a Helm release in vault-secrets-operator-system. The Helm chart deploys a VaultStaticSecret CR in the projectx namespace that maps kv/data/projectx → cloudflared-token.token. VSO creates and maintains that Kubernetes secret. When the token is rotated in Vault, VSO updates the secret within refreshAfter (1 hour) without restarting any pods.

🚀

FastAPI

Python Web Framework

Why it was invented: Python web frameworks like Flask are simple and well-known, but they lack type safety, have no built-in async support, and require separate libraries for data validation and documentation. FastAPI (Sebastián Ramírez, 2018) is built on Python type hints and Pydantic: write a function with typed parameters, and you automatically get request validation, serialisation, and an interactive Swagger docs page at /docs. It uses ASGI and async/await natively, making it one of the fastest Python frameworks.

In this project: The app in app/main.py is a minimal FastAPI service with two endpoints: GET / returns a hello-world JSON message and GET /healthz returns {"status":"ok"}. The /healthz endpoint is used by Kubernetes readiness and liveness probes to know when a pod is ready to receive traffic and when to restart it. The app runs under uvicorn on port 8000 inside the container.

📈

Horizontal Pod Autoscaler (HPA)

Auto-scaling

Why it was invented: Traffic to an application is not constant. Scaling manually — logging in and changing a replica count — wastes either money (too many pods at night) or causes outages (too few during a traffic spike). Kubernetes HPA (built-in since v1.1) watches a metric (typically CPU or memory utilisation) and automatically adds or removes pods to hit a target. Combined with cluster autoscaler (which adds/removes nodes), the entire stack scales without human involvement.

In this project: The HPA in templates/hpa.yaml keeps CPU utilisation at 70%, scaling the projectx-api Deployment between 2 and 10 replicas. During low traffic, 2 pods run (one per zone for availability). Under load, Kubernetes adds more pods automatically. Minimum 2 replicas also ensures the app survives a single node or zone going down.

Complete these steps once when starting from scratch. Run them top to bottom.

gcloud auth login

export PROJECT_ID="my-projectx-123456"
gcloud projects create $PROJECT_ID --name="ProjectX"

gcloud billing accounts list

gcloud config set project $PROJECT_ID

gcloud auth application-default login

This is separate from gcloud auth login. Terraform uses ADC, not your CLI session.

Step 6 — Grant IAM Roles

export ACCOUNT="your-email@gmail.com"
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="user:$ACCOUNT" --role="roles/owner"

Minimal roles (no owner): roles/container.admin, roles/compute.networkAdmin, roles/artifactregistry.admin, roles/iam.serviceAccountAdmin, roles/resourcemanager.projectIamAdmin, roles/serviceusage.serviceUsageAdmin, roles/iam.workloadIdentityPoolAdmin.

Step 7 — Enable Service Usage API

gcloud services enable serviceusage.googleapis.com --project=$PROJECT_ID

Step 8 — (Optional) Pre-enable APIs

Terraform does this automatically. Run only if your account cannot manage Service Usage:

gcloud services enable \
  artifactregistry.googleapis.com compute.googleapis.com \
  container.googleapis.com iam.googleapis.com \
  iamcredentials.googleapis.com --project=$PROJECT_ID

All done — continue to Deploy below.

Deploy with Terraform

Set github_repo and project ID in infra/terraform/terraform.tfvars:

gcp_project_id = "YOUR_PROJECT_ID"
github_repo    = "YOUR_USERNAME/ProjectX"
gitops_repo_url = "https://github.com/YOUR_USERNAME/ProjectX-ArgoCD.git"

Init and apply (~15 min first time):

cd ProjectX/infra/terraform
terraform init
terraform apply

Note the outputs — you'll need them for CI setup:

terraform output ci_wif_provider
terraform output ci_service_account
terraform output artifact_registry_repository

Configure kubectl:

gcloud container clusters get-credentials projectx-gke \
  --region europe-west3 --project YOUR_PROJECT_ID

Next steps in order: Build & push Docker image → Set up CI → (optional) Cloudflare tunnels → (optional) Vault → Access Argo CD

Build & Push Docker Image (Manual)

Required once before the app can be deployed. After CI is set up this runs automatically.

export PROJECT_ID="YOUR_PROJECT_ID"
export REGION="europe-west3"
export IMAGE="$REGION-docker.pkg.dev/$PROJECT_ID/projectx/projectx-api:0.1.0"

gcloud auth configure-docker "$REGION-docker.pkg.dev"
docker build -t "$IMAGE" ./app
docker push "$IMAGE"

After pushing, update image.tag in ProjectX-ArgoCD/apps/projectx-api/values.yaml and commit. ArgoCD will deploy automatically.

Run Locally

cd ProjectX/app
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload --host 0.0.0.0 --port 8000
# http://localhost:8000

CI — Automated Build & Deploy

After setup, pushing any change to app/ on main triggers the full pipeline automatically — no manual steps.

git push app/→ GitHub Actions→ WIF auth (no keys)→ docker build + push→ update image.tag→ ArgoCD sync→ new version live ✓

Total time: ~3–5 minutes from push to live.

Step 1 — Set github_repo and apply Terraform

Already done if you set github_repo = "YOUR_USERNAME/ProjectX" in terraform.tfvars and ran terraform apply. Terraform created the WIF pool, CI service account, and IAM bindings.

Step 2 — Create a GitHub PAT (for GitOps commits)

Go to github.com/settings/tokens/new → Classic token → repo scope → Generate. Copy it.

Accessing Argo CD

Get the initial admin password:

kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d && echo

kubectl -n argocd port-forward svc/argocd-server 8080:80
# http://localhost:8080  (user: admin)

Once Cloudflare tunnel is set up for ArgoCD (argocd.pbcv.dev), you can access it publicly without port-forward.

Cloudflare Tunnels

Expose both ArgoCD and the API publicly without GCP Load Balancers or open firewall ports. You create the tunnels in the Cloudflare dashboard and paste the tokens into Terraform.

argocd.pbcv.dev

Argo CD tunnel

Managed by Terraform — cloudflared Deployment in argocd namespace, points at argocd-server:80.

projectx.pbcv.dev

ProjectX API tunnel

Managed by Argo CD via Helm chart when cloudflareTunnel.enabled: true. Token injected by Terraform or Vault.

Go to one.dash.cloudflare.com → Networks → Tunnels → Create tunnel (Cloudflared).

Create tunnel 1 (ArgoCD): public hostname argocd.pbcv.dev → service http://argocd-server.argocd.svc.cluster.local:80. Copy the token.

Create tunnel 2 (API): public hostname projectx.pbcv.dev → service http://projectx-api.projectx.svc.cluster.local:80. Copy the token.

Add tokens to infra/terraform/terraform.tfvars and apply:

argocd_tunnel_token   = "eyJ..."
projectx_tunnel_token = "eyJ..."   # if not using Vault
terraform apply

Enable the tunnel in values.yaml and push (ProjectX API only — ArgoCD is Terraform-managed):

cloudflareTunnel:
  enabled: true
  hostname: projectx.pbcv.dev

If you use Vault for the ProjectX tunnel token, leave projectx_tunnel_token empty in main terraform.tfvars — Vault/VSO handles injection. See the Vault section.

HashiCorp Vault Integration

Run after terraform apply in infra/terraform/. The Vault Terraform reads the live GKE cluster.

All secrets live in one gitignored file: infra/terraform-vault/terraform.tfvars. Terraform writes them to Vault. VSO syncs them into the cluster automatically and refreshes every hour.

Step 1 — Enable KV v2 in Vault

vault secrets list
# If kv/ is not listed:
vault secrets enable -path=kv kv-v2

Step 2 — Enable VSO in Main Terraform

In infra/terraform/terraform.tfvars: install_vault_secrets_operator = true. Re-run terraform apply if needed.

Step 3 — Get Vault Token

vault login -method=userpass username=YOUR_USERNAME
# or
export VAULT_TOKEN="hvs.your_token"
vault token lookup

Step 4 — Fill in the Secrets File

cd ProjectX/infra/terraform-vault
cp terraform.tfvars.example terraform.tfvars
gcp_project_id = "YOUR_PROJECT_ID"
vault_address  = "https://vault.pbcv.dev"

cloudflare_projectx_token = "eyJ..."   # → written to kv/projectx
cloudflare_argocd_token   = "eyJ..."   # → written to kv/argocd

terraform.tfvars is gitignored — tokens never hit Git.

Step 5 — Apply

terraform init && terraform apply

Configures Vault: Kubernetes auth backend → GKE endpoint + CA cert · policy projectx-api · role projectx-api · writes both secrets.

Step 6 — Enable Vault in values.yaml

cloudflareTunnel:
  enabled: true
  hostname: projectx.pbcv.dev

vault:
  enabled: true
  address: "https://vault.pbcv.dev"
  kubernetesAuthRole: projectx-api

Push to GitHub → ArgoCD deploys VaultStaticSecret → VSO creates cloudflared-token → cloudflared starts.

Rotating a Secret

vault kv put kv/projectx token="eyJ_NEW..."
# Force immediate VSO sync:
kubectl annotate vaultstaticsecret projectx-api-cloudflared-token \
  vault.hashicorp.com/requestReconcile="$(date)" -n projectx --overwrite

cd infra/terraform-vault && terraform destroy
cd ../terraform && terraform destroy

cd ProjectX/infra/terraform-vault && terraform destroy

# Destroy all GCP infrastructure:
cd ProjectX/infra/terraform && terraform destroy

gcloud projects delete YOUR_PROJECT_ID

Deletes everything in the project. 30-day grace period to restore.

Troubleshooting

Pods in ErrImagePull / ImagePullBackOff

Fix: Terraform now grants roles/artifactregistry.reader to the GKE node SA automatically. Verify the image URL in values.yaml matches the AR repo exactly.

kubectl describe pod -n projectx -l app.kubernetes.io/name=projectx-api | grep -A5 Events

Cloudflared CrashLoopBackOff

The cloudflared-token secret doesn't exist yet. Either set projectx_tunnel_token in main terraform.tfvars and apply, or run terraform-vault if using Vault.

# Quick manual fix:
kubectl create secret generic cloudflared-token \
  --from-literal=token=YOUR_TOKEN -n projectx

GitHub Actions fails: "google-github-actions/auth: failed"

WIF variables not set or wrong. Verify:

# In GitHub repo → Settings → Variables:
WIF_PROVIDER      = projects/PROJECT_NUMBER/locations/global/workloadIdentityPools/...
GCP_SERVICE_ACCOUNT = projectx-ci@YOUR_PROJECT_ID.iam.gserviceaccount.com
GCP_PROJECT_ID    = YOUR_PROJECT_ID

Get correct values: cd infra/terraform && terraform output ci_wif_provider

VSO secret not created

kubectl describe vaultstaticsecret -n projectx
kubectl logs -n vault-secrets-operator-system \
  -l app.kubernetes.io/name=vault-secrets-operator | tail -30

Common causes: terraform-vault not applied · wrong role/SA name · KV v2 not enabled · secret path mismatch.

terraform destroy fails on Artifact Registry

Images exist from before the cleanup provisioner was added. Delete manually:

REPO="europe-west3-docker.pkg.dev/YOUR_PROJECT_ID/projectx"
gcloud artifacts docker images list "$REPO" --format="value(DIGEST)" | sort -u | \
  while read d; do gcloud artifacts docker images delete "$REPO@$d" --delete-tags --quiet; done
terraform destroy

ArgoCD Application stuck OutOfSync

kubectl describe application projectx-api -n argocd

Common causes: wrong gitops_repo_url · wrong gitops_app_path · VaultStaticSecret CRDs missing (set vault.enabled: false or install VSO).

Stale kubectl credentials

gcloud container clusters get-credentials projectx-gke \
  --region europe-west3 --project YOUR_PROJECT_ID
